# Typesense Search Setup & Architecture

## Overview
This document explains how the documentation search functionality works, how it is integrated into Docusaurus, and how to run and configure it for both local development and production environments.

## Architecture

To ensure a seamless developer experience and robust search capabilities, the local search infrastructure consists of three main components containerized via Docker Compose:

1. **Docusaurus Frontend (`docs-frontend`)**: The static site generated by Docusaurus, served via an Nginx web server.
2. **Typesense Server (`typesense-server`)**: The open-source search engine database that stores the indexed documentation text.
3. **DocSearch Scraper (`docs-scraper`)**: A Python-based crawler (using the `typesense/docsearch-scraper` Docker image) that scrapes the Docusaurus site and pushes the content into the Typesense server.

## How Local Auto-Indexing Works

1. When you run `docker-compose up -d`, Docker spins up the `typesense-server` and `docs-frontend` containers simultaneously.
2. The `Dockerfile` sets the `BUILD_TARGET=docker` environment variable. The `docusaurus.config.ts` detects this and builds the site using internal network URLs (`http://docs/`) instead of the public production URL (`https://docs.insync.pro`).
3. This ensures the output `sitemap.xml` contains internal `http://docs/` links.
4. Once the `docs-frontend` container is healthy and serving pages, the `docs-scraper` container starts.
5. The scraper reads the internal `sitemap.xml`, crawls the locally served documentation pages (via the internal Docker network), and indexes them into the `appse-ai-docs` collection inside the local Typesense server.
6. After the crawling finishes, the `docs-scraper` container exits gracefully.
7. As you type in the search bar on `http://localhost:3000`, the Docusaurus frontend queries the local `typesense-server` directly (via port `8108`) to fetch and display search results!

## Local Development Workflow

To run the full stack locally (including auto-indexing and verification):

```bash
# Safely build, start, and verify the stack
pnpm run docker:start
```

### Automated Index Verification

We have provided a built-in wrapper script that runs automatically when you execute `pnpm run docker:start`. 

If the stack boots up properly and the scraper indexes the pages successfully, the script will output `✅ Local build completed! Typesense index verified successfully.` and automatically exit safely.

If the indexer fails (returns 0 indexed documents), the script will automatically print the scraper's debug logs, tear down the docker environment, and throw a fatal error. 

If you prefer to run it manually natively via Docker, run:
```bash
docker-compose up --build -d
```
Then check the verifying container explicitly:
```bash
docker logs docs-verifier
```

If it works, it will output `SUCCESS: Index verification passed!` and show the total number of documents indexed.

You can also check the raw scraper logs if you need to debug what pages were ignored by the crawler:

```bash
docker logs docs-scraper
```

Look for an output like `Nb hits: <number>` at the bottom of the logs. If it says `0 records`, there may be a mismatch between the URLs in `sitemap.xml` and the `CONFIG` block in `docker-compose.yml`.

## Production Setup

When deploying to production, you do not use the local `docker-compose.yml` stack. The frontend, Typesense database, and scraper should be decoupled.

Follow these steps to configure search for production:

### 1. Typesense Server Deployment

Deploy the Typesense server in your production environment (e.g., via Docker on a VM, Kubernetes, or Typesense Cloud).
- Expose the Typesense API port (typically `443` for HTTPS).
- Secure the server with a strong Admin API key.
- Generate a restricted **Search-Only API Key** for the frontend.
- Assign a public domain to your Typesense server (e.g., `https://search.appse.ai`).

### 2. Update Docusaurus Configuration

In `docusaurus.config.ts`, update the `themeConfig.typesense` block to point to your production Typesense server:

```typescript
typesenseServerConfig: {
  nodes: [
    {
      host: 'search.appse.ai', // Your production Typesense hostname
      port: 443,               // Production port
      protocol: 'https',       // Production protocol
    },
  ],
  apiKey: 'YOUR_SEARCH_ONLY_API_KEY', // IMPORTANT: Use a search-only API key!
}
```

**Security Warning**: Never expose your Admin API key in `docusaurus.config.ts`. That file is bundled into the client-side JavaScript, meaning anyone could extract the key and modify your search index.

### 3. CI/CD Automated Indexing (GitHub Actions / GitLab CI)

In production, your CI/CD pipeline should run the scraper *after* successfully deploying the new documentation to your hosting provider.

Add a job to your CI/CD pipeline to execute the `typesense/docsearch-scraper` Docker image. You will provide the production configuration via environment variables.

Example CI/CD script step:

```bash
docker run -i --rm \
  -e TYPESENSE_API_KEY="YOUR_ADMIN_API_KEY" \
  -e TYPESENSE_HOST="search.appse.ai" \
  -e TYPESENSE_PORT="443" \
  -e TYPESENSE_PROTOCOL="https" \
  -e CONFIG='{"index_name":"appse-ai-docs","start_urls":["https://docs.insync.pro/"],"sitemap_urls":["https://docs.insync.pro/sitemap.xml"],"sitemap_alternate_links":true,"stop_urls":[],"selectors":{"lvl0": {"selector": "(//ul[contains(@class,'''menu__list''')]//a[contains(@class, '''menu__link menu__link--sublist menu__link--active''')]/text() | //nav[contains(@class, '''navbar''')]//a[contains(@class, '''navbar__link--active''')]/text())[last()]","type": "xpath","global": true,"default_value": "Documentation"},"lvl1":"header h1","lvl2":"article h2","lvl3":"article h3","lvl4":"article h4","lvl5":"article h5","lvl6":"article h6","text":"article p, article li"},"strip_chars":" .,;:#"}' \
  typesense/docsearch-scraper:latest
```

*Note: Ensure `start_urls` and `sitemap_urls` exactly match your production frontend URL (e.g., `https://docs.insync.pro/`).*
